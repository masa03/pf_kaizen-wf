部長承認フロー（3-3）構築手順

タスク: 3-3 ／ フロー名: 改善提案_部長承認

用途: 部長が評価・承認/差戻した際のステータス遷移・メール通知・FinalRewardAmount上書き転記

各式の「コピー」ボタンをクリックすると、改行なしでクリップボードにコピーされます。

フロー概要

[SharePoint - 項目が作成または変更されたとき]
    │  リスト: 評価データ
    │
    ▼
[条件1: 評価者種別="部長" AND 判定≠空?]
    │
    ├── No → 終了（課長評価や未確定データはスキップ）
    │
    ▼ Yes
[SharePoint - 複数の項目の取得]
    │  リスト: 改善提案メイン
    │  Filter: RequestID eq 'トリガーのRequestID'
    │
    ▼
[条件2: 判定="承認"?]
    │
    ├── Yes（承認）
    │   │
    │   ▼
    │   [メインリスト ステータス更新 + FinalRewardAmount上書き転記]
    │   │  ステータス → "承認済"
    │   │  FinalRewardAmount → 部長のRewardAmount（上書き）
    │   │
    │   ▼
    │   [Office 365 - メール送信]
    │      宛先: 申請者（To）+ 部長（CC）+ 課長（CC）
    │      テンプレート: 承認完了メール
    │
    └── No（差戻）
        │
        ▼
        [メインリスト ステータス更新]
        │  ステータス → "差戻"
        │
        ▼
        [Office 365 - メール送信]
           宛先: 申請者
           テンプレート: 差戻通知メール
3-2（課長承認）との違い: 課長承認フローでは褒賞金額≥5,000円で部長へエスカレーションする分岐があったが、部長承認フローでは金額による分岐は不要。部長に到達する時点で必ず≥5,000円のため、承認時は常にFinalRewardAmountを上書き転記して承認完了となる。

Step 1: フロー作成

  1. Power Automate → マイフロー → 新しいフロー → 自動化したクラウドフロー
  2. フロー名: 改善提案_部長承認
  3. トリガー: SharePoint - 項目が作成または変更されたとき を選択

Step 2: トリガー設定

サイトのアドレス 対象サイトURL
リスト名 評価データ
「作成または変更」を使う理由: 差戻→再提出→課長再承認→部長再評価のケースでは既存項目が更新されるため。

Step 3: 条件1 — 部長評価完了

  1. 新しいステップ → 条件
  2. 「AND」条件で2つを設定:

条件1: 評価者種別 = 部長

左辺（式タブ）
@{triggerOutputs()?['body/EvaluatorType/Value']}
演算子 次の値に等しい
右辺
部長

条件2: 判定 ≠ 空

左辺（式タブ）
@{triggerOutputs()?['body/Decision/Value']}
演算子 次の値に等しくない
右辺（式タブ）
null
判定（Decision）が空の場合は評価入力中（未確定）であるため処理しない。null チェックには「次の値に等しくない」演算子を使い、右辺に式 null を入力する。

「いいえ」の場合: 何もしない（終了）

Step 4: メインリスト取得

  1. 「はい」の中に 新しいステップ → 「SharePoint」→ 複数の項目の取得
アクション名を「メインリスト取得」にリネームすること（後続ステップの式で参照するため）。
サイトのアドレス 対象サイトURL
リスト名 改善提案メイン
フィルタークエリ
RequestID eq '@{triggerOutputs()?['body/RequestID']}'
上から順に取得 1
以降、メインリストの項目は first(body('メインリスト取得')?['value'])?['列名'] でアクセスする。

Step 5: 条件2 — 判定=承認?

  1. 新しいステップ → 条件
左辺（式タブ）
@{triggerOutputs()?['body/Decision/Value']}
演算子 次の値に等しい
右辺
承認

Step 6a: はい（承認）→ ステータス更新 + FinalRewardAmount転記 + 担当者クリア

[§7] REST APIで1アクションにまとめる。Person型列クリア・ステータス更新・金額更新を同時実行。「項目の更新」は必須列を全て記載する必要があるが、REST APIは変更するフィールドのみ指定すればよい。
  1. 新しいステップ → 「SharePoint」→ SharePoint への HTTP 要求の送信
サイトのアドレス 対象サイトURL
メソッド PATCH
URI
_api/web/lists/getbytitle('改善提案メイン')/items(@{first(body('メインリスト取得')?['value'])?['ID']})
ヘッダー
{"IF-MATCH": "*", "content-type": "application/json"}
本文
{"CurrentAssigneeEmailId": null, "CurrentAssigneeEmailStringId": null, "Status": "承認済", "FinalRewardAmount": @{triggerOutputs()?['body/RewardAmount']}}
FinalRewardAmount上書き転記: 部長のRewardAmountをメインリストのFinalRewardAmountに書き込む。課長承認時に書き込まれていた値は上書きされる（部長の評価が最終値）。
Person型列クリアは Id と StringId の両方を null に設定すること。片方だけではクリアされない場合がある。ステータスはValue形式ではなく文字列を直接指定（"Status": "承認済"）。

設計上の注意点

3-2（課長承認）との違い

課長承認フローでは褒賞金額≥5,000円で部長へエスカレーションする分岐があったが、部長承認フローでは金額による分岐は不要。部長に到達する時点で必ず≥5,000円のため、承認時は常にFinalRewardAmountを上書き転記して承認完了となる。

トリガーの条件フィルタリング

評価データリストは課長・部長の両方のデータが格納される。Step 3の条件で EvaluatorType/Value == "部長" をチェックし、課長の評価データ変更時にはこのフローが発火しないようにする。

承認完了メールの宛先

完了メールは申請者（To）+ 部長（CC）+ 課長（CC）に送信する。

差戻→再提出時の動作

部長が差戻した場合、申請者が修正・再提出すると、ステータスが「申請中」に戻り、申請通知フロー（3-1）→課長承認フロー（3-2）を経て再度部長承認フロー（3-3）に到達する。

構築後の検証（Claudeによる整合性チェック）

フロー構築完了後、以下の手順でコピペミスや構築漏れがないか検証してください。

  1. Power Automateのフローデザイナーで、冒頭の条件アクション「部長評価完了」を開く
  2. コードビュータブをクリック
  3. 表示されたJSON全体をコピー（冒頭のif文以降の全アクションが含まれる）
  4. Claudeに以下のプロンプトとともに貼り付ける:
以下はPower Automateフロー「改善提案_部長承認（3-3）」の冒頭条件アクションのコードビューJSONです。powerautomate/flow-approval-director-build.html の構築手順と比較して、式の値・列名・演算子・宛先・件名・送信元などにコピペミスや構築漏れがないか確認してください。
コードビューJSONには条件の中の全アクション（メインリスト取得、ステータス更新、メール送信等）が含まれるため、フロー全体の整合性を一度に確認できます。
