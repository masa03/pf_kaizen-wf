申請通知フロー（3-1）構築手順

タスク: 3-1 ／ フロー名: 改善WF_申請通知

用途: 申請提出時に回覧メンバー（いる場合）または承認者（課長 or 部長）へ通知メールを送信

§2更新: 1人目評価者なし（ApproverManager空）の場合、直接部長へ通知する分岐を追加

§3更新: Status=回覧中の場合に1人目の回覧メンバーへ回覧依頼メールを送信する分岐を追加

§5更新（DJ-10）: トリガーをSharePoint「項目作成時」→「Power Appsから（V2）」に変更。下書き→提出のパスでもフローが起動するようにする

各式の「コピー」ボタンをクリックすると、改行なしでクリップボードにコピーされます。

設計上の注意点

§5 DJ-10: トリガー変更の理由

下書き保存機能（§5）の導入により、申請は「新規作成→申請中」ではなく「新規作成→下書き→提出（更新）→申請中/回覧中」というパスになる。「項目が作成されたとき」トリガーでは下書き→提出パスで発火しないため、Power AppsトリガーにしてbtnSubmit.OnSelectから明示的に起動する方式に変更した。

既存フローを修正する場合: ①トリガーを「Power Apps (V2)」に変更、②直後に「改善提案メイン」を追加、③全条件・アクションの triggerOutputs()?['body/XXX'] を first(body('改善提案メイン')?['value'])?['XXX'] に書き換え

§3 回覧メンバーチェックの配置: Step 3.5 は Step 4 の前に置くこと

PA側のbtnSubmit.OnSelectでStatusを「回覧中」（回覧メンバーあり）または「申請中」（回覧メンバーなし）にPatchしてからこのフローを呼ぶ。このフロー内でStatusを見てどちらのパスに進むか判定する。Step 4（ApproverManager条件）は必ず Step 3.5 の「いいえ」ブランチ内に配置すること（Step 3.5 の外に置くと回覧中の場合も実行されてしまう）。

無限ループなし

Power Appsトリガー方式では、フロー内でステータスを更新してもフロー自身が再起動されることはない。

ステータス遷移パターン（§2・§3対応）

回覧メンバーあり: 回覧中 → 1人目の回覧メンバーへ通知（以降は回覧通知フローが連鎖）
回覧メンバーなし・1人目なし: 申請中 → 部長評価中（課長評価完全スキップ）
回覧メンバーなし・課長=申請者: 申請中 → 部長評価中（課長評価スキップ）
回覧メンバーなし・通常: 申請中 → 課長評価中 → （課長承認後に3-2フローで判定）

ApproverManager空判定の注意点

Power Apps側で1人目評価者を削除した場合、ApproverManager列に Blank() がPatchされる。「改善提案メイン」出力の ApproverManager/Email は null または空文字になる。単純な空文字比較だとnullが一致しないケースがあるため、empty() 関数（右辺: true）でnull・空文字・空配列を一括判定する。

V-1検証: 実機でApproverManager=Blank()のPatchが正常に機能し、フローが正しく1人目なし分岐に入ることを確認すること。

社員マスタ不要の理由

改善提案メインリストに ApproverManager / ApproverDirector をユーザー型で保存しているため、承認者のメールアドレスはリスト取得結果から直接取得できる。

フロー共有時の接続設定（実行のみのユーザー）

WriteSecurity=2 との併用に必須: このフローはインスタントフロー（Power Apps トリガー）のため、デフォルトでは実行ユーザーの権限で SharePoint API が呼ばれます。改善提案メインリストに WriteSecurity=2（自分のアイテムのみ編集可）が設定されている場合、申請者以外のユーザーがフローを実行すると 403 Access Denied になります。フロー所有者（サイト管理者）の接続を使用する設定に変更してください。
  1. フローの詳細ページを開く（編集画面ではなく概要ページ）
  2. 右側の 「実行のみのユーザー」 セクションの 「編集」 をクリック
  3. 「接続の使用」で SharePoint の行を確認
  4. 「この接続を使用する」（フロー所有者の接続）を選択する
    → 「実行のみのユーザーが提供」になっていると、実行ユーザーの権限で API が呼ばれ、WriteSecurity=2 で 403 エラーになる
  5. 「保存」 をクリック
対象: このフローは「項目の更新」で改善提案メインリストの Status・CurrentAssigneeEmail・EvaluationStartDate を更新するため、サイト管理者権限が必要です。
