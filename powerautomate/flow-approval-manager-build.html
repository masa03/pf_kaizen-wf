課長承認フロー（3-2）構築手順

タスク: 3-2 ／ フロー名: 改善提案_課長承認

用途: 課長が評価・承認/差戻した際のステータス遷移・メール通知・FinalRewardAmount転記

各式の「コピー」ボタンをクリックすると、改行なしでクリップボードにコピーされます。

フロー概要

[SharePoint - 項目が作成または変更されたとき]
    │  リスト: 評価データ
    │
    ▼
[条件1: 評価者種別="課長" AND 判定≠空?]
    │
    ├── No → 終了（部長評価や未確定データはスキップ）
    │
    ▼ Yes
[SharePoint - 複数の項目の取得]
    │  リスト: 改善提案メイン
    │  Filter: RequestID eq 'トリガーのRequestID'
    │
    ▼
[条件2: 判定="承認"?]
    │
    ├── Yes（承認）
    │   │
    │   ▼
    │   [条件3: RewardAmount >= 5000?]
    │       │
    │       ├── Yes（≥5,000円 → 部長承認へ）
    │       │   │
    │       │   ▼
    │       │   [メインリスト ステータス更新]
    │       │   │  ステータス → "部長評価中"
    │       │   │  ※FinalRewardAmountはまだ書き込まない
    │       │   │
    │       │   ▼
    │       │   [Office 365 - メール送信]
    │       │      宛先: ApproverDirector/Email（メインリストから取得）
    │       │      テンプレート: 承認依頼メール
    │       │
    │       └── No（<5,000円 → 承認完了）
    │           │
    │           ▼
    │           [メインリスト ステータス更新 + FinalRewardAmount転記]
    │           │  ステータス → "承認済"
    │           │  FinalRewardAmount → 課長のRewardAmount
    │           │
    │           ▼
    │           [Office 365 - メール送信]
    │              宛先: 申請者 + 課長
    │              テンプレート: 承認完了メール
    │
    └── No（差戻）
        │
        ▼
        [メインリスト ステータス更新]
        │  ステータス → "差戻"
        │
        ▼
        [Office 365 - メール送信]
           宛先: 申請者
           テンプレート: 差戻通知メール
表彰区分スキップの扱い: フロー側はRewardAmountの値のみで分岐するため、表彰区分に関わらず同一ロジックで処理できる。

Step 1: フロー作成

  1. Power Automate → マイフロー → 新しいフロー → 自動化したクラウドフロー
  2. フロー名: 改善提案_課長承認
  3. トリガー: SharePoint - 項目が作成または変更されたとき を選択

Step 2: トリガー設定

サイトのアドレス 対象サイトURL
リスト名 評価データ
「作成または変更」を使う理由: 初回評価時は項目が新規作成される。差戻→再提出後の再評価時は既存項目が更新される。両方のケースに対応するため。

Step 3: 条件1 — 課長評価完了

  1. 新しいステップ → 条件
  2. 「AND」条件で2つを設定:

条件1: 評価者種別 = 課長

左辺（式タブ）
@{triggerOutputs()?['body/EvaluatorType/Value']}
演算子 次の値に等しい
右辺
課長

条件2: 判定 ≠ 空

左辺（式タブ）
@{triggerOutputs()?['body/Decision/Value']}
演算子 次の値に等しくない
右辺（式タブ）
null
判定（Decision）が空の場合は評価入力中（未確定）であるため処理しない。

「いいえ」の場合: 何もしない（終了）

Step 4: メインリスト取得

  1. 「はい」の中に 新しいステップ → 「SharePoint」→ 複数の項目の取得
アクション名を「メインリスト取得」にリネームすること（後続ステップの式で参照するため）。
サイトのアドレス 対象サイトURL
リスト名 改善提案メイン
フィルタークエリ
RequestID eq '@{triggerOutputs()?['body/RequestID']}'
上から順に取得 1

Step 5: 条件2 — 判定=承認?

  1. 新しいステップ → 条件
左辺（式タブ）
@{triggerOutputs()?['body/Decision/Value']}
演算子 次の値に等しい
右辺
承認

Step 6a: はい（承認）→ 条件3: 金額判定

右辺
5000
右辺の 5000 は整数として入力（文字列ではない）。

設計上の注意点

トリガーの条件フィルタリング

評価データリストは課長・部長の両方のデータが格納される。Step 3の条件で EvaluatorType/Value == "課長" をチェックし、部長の評価データ変更時にはこのフローが発火しないようにする。

FinalRewardAmountの転記タイミング

≥5,000円の場合: この時点ではFinalRewardAmountを書き込まない。部長承認フロー（3-3）で部長のRewardAmountが最終値として転記される。
<5,000円の場合: 課長のRewardAmountをFinalRewardAmountに転記する。これが最終確定金額。

承認完了メールの宛先

完了メールは申請者（To）+ 課長（CC）に送信する。課長自身が承認操作を行っているため、CCで通知するだけで十分。

構築後の検証（Claudeによる整合性チェック）

フロー構築完了後、以下の手順でコピペミスや構築漏れがないか検証してください。

  1. Power Automateのフローデザイナーで、冒頭の条件アクション「課長評価完了」を開く
  2. コードビュータブをクリック
  3. 表示されたJSON全体をコピー（冒頭のif文以降の全アクションが含まれる）
  4. Claudeに以下のプロンプトとともに貼り付ける:
以下はPower Automateフロー「改善提案_課長承認（3-2）」の冒頭条件アクションのコードビューJSONです。powerautomate/flow-approval-manager-build.html の構築手順と比較して、式の値・列名・演算子・宛先・件名・送信元などにコピペミスや構築漏れがないか確認してください。
コードビューJSONには条件の中の全アクション（メインリスト取得、ステータス更新、金額判定、メール送信等）が含まれるため、フロー全体の整合性を一度に確認できます。
