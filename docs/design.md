# 1. システム概要

## 1.1 目的

業務改善提案の申請から評価・承認までのプロセスを電子化し、SharePoint + Power Platform 上で一元管理するシステムを構築する。紙ベースや既存のExcel運用を廃止し、申請状況の可視化、評価の標準化、承認プロセスの迅速化を実現する。

> ※ 給与引当処理（社員/派遣/出向の集計・ダウンロード）は本システムのスコープ外とする。

## 1.2 業務フロー概要

申請フローは以下の流れで進行する。課長承認後、褒賞金額が5,000円以上の場合のみ部長承認が必要となる条件分岐が存在する。表彰区分が「改善提案」以外（パール賞/銅賞/銀賞）の場合は、上長による評価（独創性・効果等のスコアリング）は不要だが、コメント入力等は必要なため、表彰区分に関わらず評価画面は表示される。ただしスコアリング部分は非表示とし、表彰区分のみを表示する。 なお、課長本人が申請者の場合は課長承認をスキップし、部長が第1承認者となる。

| ステップ | アクター | アクション | システム処理 |
|---------|---------|-----------|------------|
| 1 | 申請者 | 改善提案を作成・提出 | Listsに新規レコード登録、課長へメール通知 |
| 2 | 課長 | 提案内容を確認・評価・承認 | 表彰区分が改善提案の場合: 4軸評価→素点→職能換算→等級・金額を自動算出。それ以外: スコアリング不要、表彰区分に応じた金額で処理 |
| 3a | 課長 | 承認（褒賞金 < 5,000円） | ステータス更新、申請者・課長へ完了メール |
| 3b | 課長 | 承認（褒賞金 ≥ 5,000円） | ステータス更新、部長へメール通知 |
| 4 | 部長 | 提案内容を確認・再評価・承認 | 表彰区分が改善提案の場合: 4軸で再評価→等級・金額を自動算出。それ以外: スコアリング不要、承認のみ |
| 5 | 部長 | 承認 | ステータス更新、申請者・部長へ完了メール |
| ※ | 課長/部長 | 差戻（NG） | 申請者へNG通知メール、申請者が修正・再提出 |

**【重要】部長評価の最終値ルール:** 部長承認が必要なケース（褒賞金額 ≥ 5,000円）では、部長の評価結果（等級・褒賞金額）がメインリストのFinalRewardAmountに書き込まれる（部長の評価で上書き）。課長の評価データは評価データリストに参考値として保持される。部長承認が不要なケース（< 5,000円）では、課長の評価結果がFinalRewardAmountに書き込まれる。

## 1.3 表彰区分

本システムでは表彰区分として「改善提案」「パール賞」「銅賞」「銀賞」の4区分を扱う。改善提案は通常の評価フロー（4軸スコアリング）を経て等級・褒賞金額が決定される。パール賞/銅賞/銀賞は事前に別途評価済であるため、スコアリングは不要で表彰区分に応じた固定金額が適用される。

**改善提案（個人）**

| 等級 | 素点範囲 | 褒賞金額 |
|------|---------|---------|
| 6等 | 29点以下 | 300円 |
| 5等 | 30〜39点 | 600円 |
| 4等 | 40〜49点 | 1,000円 |
| 3等 | 50〜59点 | 3,000円 |
| 2等 | 60〜69点 | 5,000円 |
| 1等 | 70点以上 | 10,000円 |

**小集団（チーム）**

| 表彰区分 | 褒賞金額 | 備考 |
|---------|---------|------|
| 改善提案 | 等級に応じて決定 | 4軸評価による通常フロー |
| パール賞 | 3,000円 | 事前に別途評価済の場合のみ設定可能 |
| 銅賞 | 5,000円 | 事前に別途評価済の場合のみ設定可能 |
| 銀賞 | 10,000円 | 事前に別途評価済の場合のみ設定可能 |

**【重要】表彰区分の設定条件**

パール賞/銅賞/銀賞は、事前に別の評価プロセスで評価を受けた場合にのみ設定できる区分である。これらの表彰区分が設定されている申請については、上長（課長・部長）による独創性・効果・努力工夫・応用範囲の4軸スコアリングは不要とし、コメント入力・承認のみを行う。褒賞金額は表彰区分に応じて自動決定される。表彰区分はマスタ管理とし、区分ごとに褒賞金額を保持する。

## 1.4 利用規模

利用ユーザー数: 15,000人。改善メンバーは申請者含め最大10名。改善メンバーは別リスト（改善メンバーリスト）で管理し、申請時点での各メンバーの所属場所名を記録する。

## 1.5 GID（SonyID）について

本システムにおける「GID」は、人事マスタデータの「SonyID」に対応する10桁半角数字の社員番号である。Microsoft Entra IDとの紐付けに使用する。社員マスタリストではGID列名で統一管理する。

## 2.1 技術スタック

| コンポーネント | 技術/サービス | 用途 |
|-------------|-------------|------|
| フロントエンド | Power Apps（キャンバスアプリ） | 申請画面、評価画面、閲覧画面 |

## 2.2 データフロー

Power Apps → SharePoint Lists（データ登録/更新） → Power Automate（トリガー: リスト項目の作成/変更時） → Outlook（メール通知）の一方向フローを基本とする。評価結果による条件分岐はPower Automate内で処理し、褒賞金額に応じて次の承認者を動的に決定する。表彰区分が改善提案以外の場合はスコアリングをスキップする分岐も含む。

# 6. 評価ロジック・自動計算仕様

## 6.1 効果金額算出

改善分野ごとに実績値を入力し、分野に応じた単価または算出式で効果金額を自動計算する。合計はすべての分野の効果金額の総和とする。算出式は改善分野マスタに定義し、Power Apps側のSwitch関数で動的に処理する。

## 6.2 評価点数のスコアリング

課長・部長それぞれが4つの評価軸でスコアリングを行う。①効果については、効果金額から自動算出されるa値と、金額算定不可の場合の手動入力b値のいずれかを使用する。②〜④はラジオボタンで手動選択し、各選択肢にはあらかじめ定められた点数が対応する。

**【重要】表彰区分設定時の評価スキップ**

表彰区分（パール賞/銅賞/銀賞）が申請時に設定されている場合、上長（課長・部長）は②独創性、③努力工夫、④応用範囲の評価を行う必要はない。①効果の評価も不要。褒賞金額は表彰区分に応じた金額（パール:3,000円、銅:5,000円、銀:10,000円）が自動適用される。上長は申請内容を確認し、承認または差戻のみ行う。

## 6.3 等級判定フロー

素点合計 = ①(a or b) + ② + ③ + ④ を算出後、職能換算テーブルで調整済みスコアに変換する。最終的に表彰区分（改善提案/小集団）に応じた等級テーブルで等級を判定し、褒賞金額を確定する。褒賞金額が5,000円以上の場合、部長承認フローへ自動遷移する。表彰区分が設定されている場合はこの等級判定フローをスキップし、賞区分の金額をそのまま使用する。

**【重要】FinalRewardAmountの転記ルール:**
- 課長承認のみで完結する場合（< 5,000円）: 課長の評価データのRewardAmountをメインリストのFinalRewardAmountに転記
- 部長承認が必要な場合（≥ 5,000円）: 部長の評価データのRewardAmountをメインリストのFinalRewardAmountに転記（部長の評価で上書き）
- 課長の評価データは評価データリストにそのまま保持（削除・変更しない）

## 6.4 条件分岐ロジック

| 条件 | 分岐先 | 処理 |
|------|-------|------|
| 課長承認 AND 褒賞金額 < 5,000円 | 承認完了 | ステータス→承認済、課長の褒賞金額→FinalRewardAmount転記、申請者+課長へ完了メール |
| 課長承認 AND 褒賞金額 ≥ 5,000円 | 部長評価 | ステータス→部長評価中、部長通知 |
| 課長差戻 | 申請者修正 | ステータス→差戻、NG通知 |
| 部長承認 | 承認完了 | ステータス→承認済、部長の褒賞金額→FinalRewardAmount転記（上書き）、申請者+部長へ完了メール |
| 部長差戻 | 申請者修正 | ステータス→差戻、NG通知 |
| 表彰区分=パール賞/銅賞/銀賞 | 表彰区分金額で決定（スコアリングスキップ） | 表彰区分マスタから金額取得。パール:3,000/銅:5,000/銀:10,000 |
| 差戻→再提出 | 課長評価から再開 | ステータス→申請中に戻し課長へ再通知。前回評価データは参考値として保持 |

# 7. セキュリティ・権限設計

## 7.1 ロール定義

> ※ 承認者マスタでの指定から、社員マスタの本人フラグでのロール判定に変更。

| ロール | 対象者 | 権限 |
|-------|-------|------|
| 申請者 | 全社員（15,000人） | 自分の申請の作成・編集・閲覧 |
| 課長 | 社員マスタの課長本人フラグ=1 | 担当部署の申請の閲覧・評価・承認/差戻 |
| 部長 | 社員マスタの部長本人フラグ=1 | 担当部署の申請の閲覧・評価・承認/差戻（≥5,000円） |
| 管理者 | システム管理者 | 全件閲覧・マスタメンテナンス |

## 7.2 データアクセス制御

SharePoint Listsのアイテムレベル権限を使用し、申請者は自分の申請のみ編集可能とする。承認者は評価データリストの自分のレコードのみ編集可能とする。Power Apps側でもフィルタリングを行い、ロールに応じた画面表示制御を実装する。マスタリストは管理者のみ編集可能とし、他ユーザーは読み取りのみとする。申請内容閲覧画面は、申請者本人および承認者（課長・部長）がアクセス可能とする。

## 7.3 テストモード仕様

テスト環境ではPower Platformを利用可能なアカウントが1つ（m.kato@...）のみであるため、1アカウントで申請者・課長・部長の全ロールをテストできるようテストモードを導入する。本番デプロイ時にテストモードをOFFにすることで、本番仕様に切り替わる。

**モード切替**

アプリ内にテストモードフラグ（グローバル変数）を設け、ON/OFFで動作を切り替える。

| 項目 | テストモード（ON） | 本番モード（OFF） |
|------|-------------------|-------------------|
| ユーザー特定方式 | GIDをテキスト入力 → 社員マスタLookUp → その人として全処理 | User().Email → 社員マスタLookUp → 自動特定 |
| User型フィールド | 全レコード同一アカウント（m.kato@...）が入る | 実際のユーザーアカウントで解決 |
| メール通知宛先 | 全て自分（m.kato@...）に届く | 社員マスタの課長/部長メールアドレス宛 |
| メール本文 | 本来の宛先を明記（例:「宛先: 田中太郎（課長）」） | 通常通り |

**テスト時の運用**

- GID入力欄に任意の社員GIDを入力することで、その社員として申請・承認操作を実行できる
- 課長承認テスト時: 課長のGIDを入力して評価画面を操作
- 部長承認テスト時: 部長のGIDを入力して評価画面を操作
- User型フィールドは全レコード同じアカウントになるが、テスト段階では支障なし
- メール通知は全て自分に届くが、本文中の宛先情報で誰宛かを判別する

**本番切替時の対応**

- テストモードフラグをOFFに変更
- GID入力欄を非表示にし、User().Emailからの自動取得に切替
- User型フィールド・メール通知は実アカウントで動作
