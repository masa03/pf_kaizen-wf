# 7. セキュリティ・権限設計

## 7.1 ロール定義

> ※ 承認者マスタでの指定から、社員マスタの本人フラグでのロール判定に変更。

| ロール | 対象者 | 権限 |
|-------|-------|------|
| 申請者 | 全社員（15,000人） | 自分の申請の作成・編集・閲覧 |
| 課長 | 社員マスタの課長本人フラグ=1 | 担当部署の申請の閲覧・評価・承認/差戻 |
| 部長 | 社員マスタの部長本人フラグ=1 | 担当部署の申請の閲覧・評価・承認/差戻（≥5,000円） |
| 管理者 | システム管理者 | 全件閲覧・マスタメンテナンス |

## 7.2 データアクセス制御

SharePoint Listsのアイテムレベル権限を使用し、申請者は自分の申請のみ編集可能とする。承認者は評価データリストの自分のレコードのみ編集可能とする。Power Apps側でもフィルタリングを行い、ロールに応じた画面表示制御を実装する。マスタリストは管理者のみ編集可能とし、他ユーザーは読み取りのみとする。申請内容閲覧画面は、申請者本人および承認者（課長・部長）がアクセス可能とする。

## 7.3 テストモード仕様

テスト環境ではPower Platformを利用可能なアカウントが1つ（m.kato@...）のみであるため、1アカウントで申請者・課長・部長の全ロールをテストできるようテストモードを導入する。本番デプロイ時にテストモードをOFFにすることで、本番仕様に切り替わる。

**モード切替**

アプリ内にテストモードフラグ（グローバル変数）を設け、ON/OFFで動作を切り替える。

| 項目 | テストモード（ON） | 本番モード（OFF） |
|------|-------------------|-------------------|
| ユーザー特定方式 | GIDをテキスト入力 → 社員マスタLookUp → その人として全処理 | User().Email → 社員マスタLookUp → 自動特定 |
| User型フィールド | 全レコード同一アカウント（m.kato@...）が入る | 実際のユーザーアカウントで解決 |
| メール通知宛先 | 全て自分（m.kato@...）に届く | 社員マスタの課長/部長メールアドレス宛 |
| メール本文 | 本来の宛先を明記（例:「宛先: 田中太郎（課長）」） | 通常通り |

**テスト時の運用**

- GID入力欄に任意の社員GIDを入力することで、その社員として申請・承認操作を実行できる
- 課長承認テスト時: 課長のGIDを入力して評価画面を操作
- 部長承認テスト時: 部長のGIDを入力して評価画面を操作
- User型フィールドは全レコード同じアカウントになるが、テスト段階では支障なし
- メール通知は全て自分に届くが、本文中の宛先情報で誰宛かを判別する

**本番切替時の対応**

- テストモードフラグをOFFに変更
- GID入力欄を非表示にし、User().Emailからの自動取得に切替
- User型フィールド・メール通知は実アカウントで動作
