# 7. セキュリティ・権限設計

## 7.1 ロール定義

> ※ 承認者マスタでの指定から、社員マスタの本人フラグでのロール判定に変更。

| ロール | 対象者 | 権限 |
|-------|-------|------|
| 申請者 | 全社員（15,000人） | 自分の申請の作成・編集・閲覧 |
| 課長 | 社員マスタの課長本人フラグ=1 | 担当部署の申請の閲覧・評価・承認/差戻 |
| 部長 | 社員マスタの部長本人フラグ=1 | 担当部署の申請の閲覧・評価・承認/差戻（≥5,000円） |
| 管理者 | システム管理者 | 全件閲覧・マスタメンテナンス |

## 7.2 データアクセス制御

### 権限制御の2層構造

データの改ざん防止は、SharePointのネイティブ権限（第1層）とPower Appsのアプリロジック（第2層）の2層で実現する。

**第1層: SharePointアイテムレベル権限（バイパス不可）**

- トランザクションリスト（改善提案メイン、改善メンバー、改善分野実績、評価データ）は `WriteSecurity=2`（自分が作成したアイテムのみ編集可）を設定し、第三者による改ざんをSPプラットフォームレベルで防止する
- マスタリスト（社員マスタ、改善分野マスタ、表彰区分マスタ）はメンバーグループの権限を「読み取り」に降格し、管理者のみ編集可とする

**第1.5層: サイトナビゲーション非表示（カジュアルアクセス抑止）**

`WriteSecurity=2` は他人の改ざんを防止するが、本人がSPリストを直接開いてステータス無視で編集することは防げない（SPプラットフォームの制限）。これを抑止するため、管理者が直接参照する必要がないリストはサイトナビゲーションから非表示にする。

| リスト | ナビゲーション | 理由 |
|---|---|---|
| 改善提案メイン | 表示 | Column Formattingリンク、管理者の申請一覧確認で使用 |
| 社員マスタ | 表示 | 管理者がメンテナンスで直接編集 |
| 改善分野マスタ / 表彰区分マスタ | 表示 | 管理者がメンテナンスで参照 |
| 改善メンバー | **非表示** | Power Apps経由でのみ操作 |
| 改善分野実績 | **非表示** | Power Apps経由でのみ操作 |
| 評価データ | **非表示** | Power Apps経由でのみ操作。報奨金に直結 |
| 回覧メンバー | **非表示** | Power Apps経由でのみ操作 |
| 添付ファイル | **非表示** | Power Apps経由でのみ参照 |

> ナビゲーション非表示でもURL直打ちでのアクセスは可能。管理者のトラブルシューティングには支障なし。サイト設定 → 詳細設定から表示に戻せる。

**第2層: Power Appsアプリロジック（UI制御）**

- ロールに応じた画面表示制御（課長/部長の判定は社員マスタのフラグで実施）
- 申請フォームの編集は、ステータスが「下書き」「差戻」「取下げ」の場合のみ許可
- 評価画面のアクセスは、自分が評価者として割り当てられた申請のみ
- 申請内容閲覧画面は、申請者本人および承認者（課長・部長）がアクセス可能

**第2層補足: 評価者本人チェック（なりすまし防止）**

評価画面のOnVisibleで `User().Email` と申請の評価者メールアドレスを照合し、不一致の場合は評価フォームを非表示にしてエラーメッセージを表示する。URLパラメータを知っているだけでは評価操作できないようにする。

| モード | チェック対象 |
|---|---|
| 課長評価 | `User().Email` = `改善提案メイン.ApproverManager.Email` |
| 部長評価 | `User().Email` = `改善提案メイン.ApproverDirector.Email` |
| 回覧者 | `User().Email` が `回覧メンバー` リストに存在するか |
| テストモード | チェックをスキップ（1アカウントで全ロールテストするため） |

### アイテムレベル権限（WriteSecurity）一覧

| リスト | ReadSecurity | WriteSecurity | 理由 |
|---|---|---|---|
| 改善提案メイン | 1（全件読み取り可） | 2（自分のアイテムのみ編集） | 課長/部長が閲覧する必要があるため読み取りは全件。編集は申請者本人のみ |
| 改善メンバー | 1（全件読み取り可） | 2（自分のアイテムのみ編集） | 申請者がPatchで作成するため、Created By=申請者本人。他人のメンバーデータを改ざん防止 |
| 改善分野実績 | 1（全件読み取り可） | 2（自分のアイテムのみ編集） | 申請者がPatchで作成するため、Created By=申請者本人。他人の実績データを改ざん防止 |
| 評価データ | 1（全件読み取り可） | 2（自分のアイテムのみ編集） | 評価者本人がPower Appsから直接Patchするため、Created By=評価者本人。報奨金に直結するデータのため、第三者による改ざんをSPレベルで防止 |
| マスタ3リスト | 1（全件読み取り可） | —（リストレベルで編集権限を読み取りに降格） | 管理者のみ編集可 |

## 7.3 テストモード仕様

テスト環境ではPower Platformを利用可能なアカウントが1つ（m.kato@...）のみであるため、1アカウントで申請者・課長・部長の全ロールをテストできるようテストモードを導入する。本番デプロイ時にテストモードをOFFにすることで、本番仕様に切り替わる。

**モード切替**

アプリ内にテストモードフラグ（グローバル変数）を設け、ON/OFFで動作を切り替える。

| 項目 | テストモード（ON） | 本番モード（OFF） |
|------|-------------------|-------------------|
| ユーザー特定方式 | GIDをテキスト入力 → 社員マスタLookUp → その人として全処理 | User().Email → 社員マスタLookUp → 自動特定 |
| User型フィールド | 全レコード同一アカウント（m.kato@...）が入る | 実際のユーザーアカウントで解決 |
| メール通知宛先 | 全て自分（m.kato@...）に届く | 社員マスタの課長/部長メールアドレス宛 |
| メール本文 | 本来の宛先を明記（例:「宛先: 田中太郎（課長）」） | 通常通り |

**テスト時の運用**

- GID入力欄に任意の社員GIDを入力することで、その社員として申請・承認操作を実行できる
- 課長承認テスト時: 課長のGIDを入力して評価画面を操作
- 部長承認テスト時: 部長のGIDを入力して評価画面を操作
- User型フィールドは全レコード同じアカウントになるが、テスト段階では支障なし
- メール通知は全て自分に届くが、本文中の宛先情報で誰宛かを判別する

**本番切替時の対応**

- テストモードフラグをOFFに変更
- GID入力欄を非表示にし、User().Emailからの自動取得に切替
- User型フィールド・メール通知は実アカウントで動作
