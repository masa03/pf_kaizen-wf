> - 効果金額 = 仕掛かり金額（WIPAmount）そのまま。実績値との乗算は行わない
> - 実績値（分/月）は入力欄を表示し、ActualValueに参考記録として保存する（計算には使用しない）

**右カラム - 改善メンバー**

> ※ v5からの変更: メンバー情報は改善メンバーリスト（別テーブル）で管理。各メンバーのGID入力時に社員マスタから氏名・所属場所を自動取得し、申請時点の情報を改善メンバーリストに保存する。

> ※ §8からの変更: GID手入力方式からサジェスト検索方式に変更。氏名またはGIDを入力すると社員マスタを検索し、候補一覧から選択してメンバーに追加する。

**改善メンバー追加UIパーツ（社員サジェスト検索）** `[§8]`

```
cntMemberSearch (縦・AutoLayout)
  ├── txtMemberSearch   … テキスト入力（氏名 or GID、プレースホルダー: 「氏名またはGIDを入力」）
  └── galMemberSuggest  … サジェストリスト（最大10件、候補あり時のみ表示）
        └── cntMemberSuggestRow (横・テンプレート行、36px)
              ├── lblMemberSuggestName  … 氏名（FillPortions=1、size=12）
              ├── lblMemberSuggestBu    … 部（幅80、size=11、グレー）
              └── lblMemberSuggestGID   … GID（幅90、size=10、グレー）
```

**検索ロジック**: 入力値が全数字 → GID完全一致検索（`Filter(社員マスタ, GID = 入力値 And IsActive = true)`）、それ以外 → 氏名前方一致検索（`Filter(社員マスタ, IsActive = true, StartsWith(EmployeeName, 入力値))`）。いずれも最大10件。

**候補選択時**: 重複チェック（MemberGID比較）・上限チェック（10名）を行い、`colMembers` に `{MemberGID, MemberName, MemberOffice, MemberCostUnit, SortOrder}` を追加。`txtMemberSearch` をリセット。

|------|---------|----------------|
| メンバー追加 | サジェスト検索 `[§8]` | 氏名またはGIDを入力→候補選択。退職者（IsActive=false）は除外。上限10名 |

## 4.4 評価画面詳細（課長/部長共通）

## 4.5 申請状況確認導線 `[v10.1]`

申請者が自分の申請一覧と評価結果を確認するための導線。SharePointリストのカスタムビュー＋Column Formattingで実現し、Power Appsホーム画面の開発を不要とする。

### 4.5.1 SharePointリスト カスタムビュー

改善提案メインリストに以下のカスタムビューを作成する。

**ビュー: 自分の申請**

| 設定項目 | 内容 |
|---------|------|
| ビュー名 | 自分の申請 |
| フィルタ | ApplicantEmail = [Me] |
| 並び替え | ID 降順（新しい順） |
| 表示列 | RequestID / Theme / Status / CompletionDate / FinalRewardAmount |

**ビュー: 自分の承認待ち** `[v2]`

| 設定項目 | 内容 |
|---------|------|
| ビュー名 | 自分の承認待ち |
| フィルタ | CurrentAssigneeEmail = [Me] |
| 並び替え | ID 降順（新しい順） |

> CurrentAssigneeEmail（Person型）には、評価者（課長・部長）および回覧者のメールアドレスがセットされる。承認完了・差戻・取消時は空にクリアされるため、アクションすべき案件のみが表示される。

### 4.5.2 Column Formatting（列の書式設定）

#### 列レベル書式（全ビュー共通）

RequestID列にJSON書式設定を適用し、Power Apps閲覧画面へのリンクボタンを表示する。「自分の申請」「すべてのアイテム」ビューで使用される。

```json
{
  "$schema": "https://columnformatting.sharepointpnp.com/columnFormattingSchema.json",
  "elmType": "a",
  "txtContent": "@currentField",
  "style": {
    "color": "#0078d4",
    "text-decoration": "underline",
    "cursor": "pointer"
  },
  "attributes": {
    "href": "='https://apps.powerapps.com/play/{AppID}?RequestID=' + @currentField",
    "target": "_blank"
  }
}
```

> **注意**: `{AppID}` はPower AppsアプリのGUIDに置換すること。アプリ公開後に確定する。

#### ビューレベル書式：「自分の承認待ち」ビュー `[§13]`

「自分の承認待ち」ビューでは、列レベル書式をビューレベルで上書きし、Status列の値に基づいて承認画面へ直接遷移するパラメータを付与する。

| Status | 付与パラメータ | 遷移先 |
|--------|--------------|--------|
| 課長評価中 | `&EvalType=課長` | 評価画面（課長モード） |
| 部長評価中 | `&EvalType=部長` | 評価画面（部長モード） |
| 回覧中 | `&Mode=Reviewer` | 評価画面（回覧モード） |
| その他 | なし | 閲覧画面 |

```json
// ビューの CustomFormatter に設定（キーは列内部名）
{
  "RequestID": {
    "elmType": "a",
    "txtContent": "@currentField",
    "style": { "color": "#0078d4", "text-decoration": "underline", "cursor": "pointer" },
    "attributes": {
      "href": "='https://apps.powerapps.com/play/{AppID}?RequestID=' + @currentField + if([$Status] == '課長評価中', '&EvalType=課長', if([$Status] == '部長評価中', '&EvalType=部長', if([$Status] == '回覧中', '&Mode=Reviewer', '')))",
      "target": "_blank"
    }
  }
}
```

> Power Apps側の対応は不要。App.OnStart / App.StartScreen は既に `EvalType` / `Mode=Reviewer` パラメータに対応済み。

### 4.5.3 Power Apps URLパラメータ受け取り

App.OnStartにURLパラメータ処理を追加し、メールリンクやSharePointからの直接遷移を可能にする。

```
// URLパラメータで閲覧画面に直接遷移
If(
    !IsBlank(Param("RequestID")),
    Set(varViewRequestID, Param("RequestID"));
    Set(varViewMode, "View");
    Navigate(ViewScreen)
)
```

### 4.5.4 ユーザー導線

| 導線 | フロー |
|------|--------|
| SharePointから（自分の申請） | 改善提案メインリスト → 「自分の申請」ビュー → RequestIDクリック → Power Apps閲覧画面 |
| SharePointから（承認待ち） | 改善提案メインリスト → 「自分の承認待ち」ビュー → RequestIDクリック → Power Apps評価画面（Status に応じた画面） `[§13]` |

| アプリ内から | 申請フォームのプレビューボタン → 閲覧画面（従来通り） |

## 4.8 v2 追加機能 — 画面設計変更

v2で追加された各機能による画面設計の変更点をまとめる。各機能の詳細な変更提案原本は `docs/changes/archive/` に保存されている。

### 4.8.1 §1 添付ファイル多形式対応 `[v2]`

v1の `AddMediaButton`（画像ファイルのみ選択可）を廃止し、EditForm の Attachment コントロール＋添付ファイルステージングリスト方式に変更。PDF・PowerPoint・Excel・Word等の非画像ファイルもアップロード可能にする。

**申請フォームの変更:**

- `AddMediaButton` / `UploadedImage1` を削除し、`editFormAttachment`（EditForm）に置換
- カテゴリドロップダウン（`ddAttachFileCategory`）でカテゴリ（改善前/改善後/その他）を選択 → EditForm で対応するステージングレコードに切り替え → 「このカテゴリを確定」ボタン（`btnConfirmAttachment`）で `SubmitForm()` 実行
- 容量注記ラベル（`lblFileNote`）: 「対応確認済み形式: JPG, PNG, GIF, PDF, PPTX, XLSX, DOCX（目安: 1ファイル30MB以内）」を表示
- 確定済みファイル一覧ギャラリー（`galStagingFiles`）: `colStagingDisplay` の内容を表示
- OnVisible でステージングレコード3件（改善前/改善後/その他）を自動作成
- キャンセル時・Navigate前にステージングレコードを削除（クリーンアップ処理）
- プレビュー時の添付ファイル表示: 画像・非画像問わず「（添付済み: N件）」テキスト表示のみ

**閲覧画面の変更:**

- 画像ファイル（JPG/JPEG/PNG/GIF）: 従来通り Image コントロールで埋め込み表示
- 非画像ファイル（PDF/PPTX/XLSX/DOCX等）: ファイル名リンク表示（`Launch()` で SharePoint ビューアを開く）
- 改善前/改善後に画像ファイルがない場合: 「画像なし」プレースホルダー表示（非画像ファイルがある場合はリンク表示）

**フロー連携:**

- 提出時に `改善WF_ステージング転送.Run()` でステージングレコードからドキュメントライブラリへ転送
- `改善WF_ステージングクリーンアップ`（日次スケジュール）で48時間以上経過した未使用ステージングレコードを自動削除
- 旧 `改善WF_添付ファイルアップロード` は廃止（既存データ対応のためフロー自体は残存）

> **変更提案原本**: `docs/changes/archive/v2-file-format/proposal.md`

### 4.8.2 §2 評価者変更機能 `[v2]`

申請フォームで評価者（1人目=課長相当、2人目=部長相当）を表示・変更・削除できる機能を追加する。

**申請フォームの変更:**

- 右カラムに評価者セクションを追加（回覧者セクション → 評価者セクション → 改善メンバーセクションの順）
- 1人目の評価者: デフォルトは社員マスタの `ManagerGID` から自動取得。GID入力で変更可能。削除可（なし設定可）
- 2人目の評価者: デフォルトは社員マスタの `DirectorGID` から自動取得。GID入力で変更可能。削除不可（必須）
- `ManagerGID` が空の場合: 1人目は最初から「なし」状態で表示
- バリデーション: 申請者自身の設定禁止、1人目と2人目の重複禁止
- 差戻再編集時: メインリストの `ApproverManager` / `ApproverDirector` から復元

**閲覧画面の変更:**

- 承認者情報セクション: 1人目なし時に「（なし）」を表示
- 1人目なし時: 評価結果（課長）セクションを非表示

**ステータス遷移への影響:**

- 1人目なし: 「申請中」→「部長評価中」に直接遷移（課長評価中をスキップ）
- 1人目なし時の金額分岐: スキップ（2人目の承認で即完了）

> **変更提案原本**: `docs/changes/archive/v2-evaluator-change/proposal.md`

### 4.8.3 §3 回覧者機能 `[v2]`

評価者（課長・部長）に回る前に、申請内容を事前確認する「回覧者」を追加できる機能。回覧者はスコアリング不要、承認/差戻のみ。最大5名、直列（順番に）回覧する。

**申請フォームの変更:**

- 右カラムに回覧者セクションを追加（デフォルト折りたたみ）
- サジェスト検索UI（§8）で社員マスタから選択して回覧者を追加
- 上下ボタンで回覧順序の並べ替えが可能
- バリデーション: 申請者自身の指定禁止、評価者（課長・部長）との重複禁止、同一人物の重複指定不可、最大5名
- 提出時にコレクション `colReviewers` を回覧者リスト（SharePoint）に書き込み
- 差戻/取下げ後の再編集時: 既存の回覧者リストから `colReviewers` に復元、編集も可能

**評価画面の変更（Reviewerモード追加）:**

| モード | URLパラメータ | 表示内容 |
|--------|-------------|---------|
| 課長評価（既存） | `EvalType=課長` | 閲覧画面（上部）＋ スコアリング ＋ 承認/差戻ボタン |
| 部長評価（既存） | `EvalType=部長` | 閲覧画面（上部）＋ スコアリング ＋ 承認/差戻ボタン |
| **回覧者（新規）** | `Mode=Reviewer` | 閲覧画面（上部）＋ 差戻コメント入力 ＋ 承認/差戻ボタンのみ |

- Reviewerモード時: スコアリング部分（①〜④評価軸、素点合計、職能換算、等級、褒賞金額、おすすめ情報）は非表示
- 差戻時: コメント入力必須。承認時はコメント入力不要
- 差戻コメントは SP 非保存（メール通知本文にのみ記載）

**閲覧画面の変更:**

- 改善メンバー一覧の直下に回覧者一覧セクションを追加（回覧順、氏名、ステータス表示）
- 回覧者が0名の場合はセクション非表示

> **変更提案原本**: `docs/changes/archive/v2-reviewer/proposal.md`

### 4.8.4 §4 申請取消機能 `[v2]`

申請者が回覧中・評価待ちの申請を自ら取り消せる機能。v1の取下げ（下書き・申請中のみ）を拡張し、「回覧中」「課長評価中」「部長評価中」からも取消可能にする。

**閲覧画面の変更:**

- アクション領域（ステータスバッジ・承認者情報の下）に取消ボタンを追加
- 取消ボタンの表示条件: 申請者本人 AND 取消可能ステータス（下書き/申請中/回覧中/課長評価中/部長評価中） AND Viewモード
- Embeddedモード（評価画面に組み込まれた閲覧画面）では取消ボタン非表示
- 確認ポップアップ: 取消理由テキストボックス（任意入力）＋「取消を実行する」「キャンセル」ボタン

**取消実行処理:**

1. ステータスを「取下げ」に Patch 更新
2. 取下げ通知フロー（No.4）を Power Apps から直接起動（下書きの場合はスキップ）
3. 取消理由・取消前ステータスをフローにパラメータ渡し
4. 取消理由は SP 非保存（メール通知本文にのみ記載、空の場合は「（理由なし）」表示）

**取消後:**

- 差戻と同じ導線で再編集・再提出が可能（ステータス「取下げ」のレコードを編集モードで開ける）
- 既存の評価データは保持（削除しない）

> **変更提案原本**: `docs/changes/archive/v2-cancel/proposal.md`

### 4.8.5 §5 下書き保存機能 `[v2]`

申請フォームで入力途中の内容を一時保存（下書き保存）できる機能。

**申請フォームの変更:**

- 提出ボタンの左に「下書き保存」ボタンを追加（セカンダリスタイル）
- 下書き保存時のバリデーション: 表彰区分のみ必須、他の入力項目はスキップ
- SP側必須列でユーザー手入力の列（Theme, Problem, Improvement等）は空文字列 `""` で保存
- CompletionDate 未入力時は `Today()` で仮保存
- 子リスト（改善メンバー・改善分野実績）も全件保存（提出時と同じ全件削除→新規作成ロジックを再利用）
- 添付ファイルも下書き保存時にアップロード
- 保存完了時は `Notify("下書きを保存しました")` のみ表示（画面遷移なし、編集継続可能）
- 差戻状態からの下書き保存も可能（ステータスを「下書き」に変更）

**RequestID 採番方式の変更:**

- Power Apps 側で KZ 形式を直接生成: `"KZ-" & Text(Year(Now())) & "-" & Text(varNewRequest.ID, "00000")`
- 採番タイミング: 初回 Patch 直後（下書き保存 or 新規提出）
- 下書き状態でも正式な KZ 形式の RequestID が付与される

**閲覧画面の変更:**

- 「編集」ボタンを追加: 申請者本人 AND ステータスが下書き/差戻/取下げの場合のみ表示
- 「編集」ボタン押下 → 申請フォーム画面に `Mode=Edit`, `RequestID` 付きで遷移

**申請通知フローのトリガー変更:**

- 「Lists 項目作成時」→「Power Apps から直接起動」に変更
- 下書き→再提出時にフロー直接起動（`申請通知フロー.Run(varEditRequestID)`）
- 回覧者ありの場合は回覧通知フローを起動、なしの場合は申請通知フローを起動

> **変更提案原本**: `docs/changes/archive/v2-draft-save/proposal.md`

### 4.8.6 §7 申請状況確認導線 `[v2]`

SharePoint リストのカスタムビュー＋ Column Formatting により、承認者・回覧者が自分の対応待ち案件を確認できる「自分の承認待ち」ビューを追加する。

**Column Formatting:**

- §13 で実装済み。「自分の承認待ち」ビューではビューレベル書式で Status 列に基づく遷移パラメータを付与（4.5.2節参照）
