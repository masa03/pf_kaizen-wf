# 1. システム概要

## 1.1 目的

業務改善提案の申請から評価・承認までのプロセスを電子化し、SharePoint + Power Platform 上で一元管理するシステムを構築する。紙ベースや既存のExcel運用を廃止し、申請状況の可視化、評価の標準化、承認プロセスの迅速化を実現する。

> ※ 給与引当処理（社員/派遣/出向の集計・ダウンロード）は本システムのスコープ外とする。

## 1.2 業務フロー概要

申請フローは以下の流れで進行する。課長承認後、褒賞金額が5,000円以上の場合のみ部長承認が必要となる条件分岐が存在する。表彰区分が「改善提案」以外（小集団 パール賞（3等）/小集団 銅賞（2等）/小集団 銀賞（1等））の場合は、上長による評価（独創性・効果等のスコアリング）は不要だが、コメント入力等は必要なため、表彰区分に関わらず評価画面は表示される。ただしスコアリング部分は非表示とし、表彰区分のみを表示する。 なお、課長本人が申請者の場合は課長承認をスキップし、部長が第1承認者となる。

| ステップ | アクター | アクション | システム処理 |
|---------|---------|-----------|------------|
| 1 | 申請者 | 改善提案を作成・提出 | Listsに新規レコード登録、課長へメール通知 |
| 2 | 課長 | 提案内容を確認・評価・承認 | 表彰区分が改善提案の場合: 4軸評価→素点→職能換算→等級・金額を自動算出。それ以外: スコアリング不要、表彰区分に応じた金額で処理 |
| 3a | 課長 | 承認（褒賞金 < 5,000円） | ステータス更新、申請者・課長へ完了メール |
| 3b | 課長 | 承認（褒賞金 ≥ 5,000円） | ステータス更新、部長へメール通知 |
| 4 | 部長 | 提案内容を確認・再評価・承認 | 表彰区分が改善提案の場合: 4軸で再評価→等級・金額を自動算出。それ以外: スコアリング不要、承認のみ |
| 5 | 部長 | 承認 | ステータス更新、申請者・部長へ完了メール |
| ※ | 課長/部長 | 差戻（NG） | 申請者へNG通知メール、申請者が修正・再提出 |

**【重要】部長評価の最終値ルール:** 部長承認が必要なケース（褒賞金額 ≥ 5,000円）では、部長の評価結果（等級・褒賞金額）がメインリストのFinalRewardAmountに書き込まれる（部長の評価で上書き）。課長の評価データは評価データリストに参考値として保持される。部長承認が不要なケース（< 5,000円）では、課長の評価結果がFinalRewardAmountに書き込まれる。

## 1.3 表彰区分

本システムでは表彰区分として「改善提案」「小集団 パール賞（3等）」「小集団 銅賞（2等）」「小集団 銀賞（1等）」の4区分を扱う。改善提案は通常の評価フロー（4軸スコアリング）を経て等級・褒賞金額が決定される。小集団 パール賞（3等）/小集団 銅賞（2等）/小集団 銀賞（1等）は事前に別途評価済であるため、スコアリングは不要で表彰区分に応じた固定金額が適用される。

**改善提案（個人）**

| 等級 | 素点範囲 | 褒賞金額 |
|------|---------|---------|
| 6等 | 29点以下 | 300円 |
| 5等 | 30〜39点 | 600円 |
| 4等 | 40〜49点 | 1,000円 |
| 3等 | 50〜59点 | 3,000円 |
| 2等 | 60〜69点 | 5,000円 |
| 1等 | 70点以上 | 10,000円 |

**小集団（チーム）**

| 表彰区分 | 褒賞金額 | 備考 |
|---------|---------|------|
| 改善提案 | 等級に応じて決定 | 4軸評価による通常フロー |
| 小集団 パール賞（3等） | 3,000円 | 事前に別途評価済の場合のみ設定可能 |
| 小集団 銅賞（2等） | 5,000円 | 事前に別途評価済の場合のみ設定可能 |
| 小集団 銀賞（1等） | 10,000円 | 事前に別途評価済の場合のみ設定可能 |

**【重要】表彰区分の設定条件**

小集団 パール賞（3等）/小集団 銅賞（2等）/小集団 銀賞（1等）は、事前に別の評価プロセスで評価を受けた場合にのみ設定できる区分である。これらの表彰区分が設定されている申請については、上長（課長・部長）による独創性・効果・努力工夫・応用範囲の4軸スコアリングは不要とし、コメント入力・承認のみを行う。褒賞金額は表彰区分に応じて自動決定される。表彰区分はマスタ管理とし、区分ごとに褒賞金額を保持する。

## 1.4 利用規模

利用ユーザー数: 15,000人。改善メンバーは申請者含め最大10名。改善メンバーは別リスト（改善メンバーリスト）で管理し、申請時点での各メンバーの所属場所名を記録する。

## 1.5 GID（SonyID）について

本システムにおける「GID」は、人事マスタデータの「SonyID」に対応する10桁半角数字の社員番号である。Microsoft Entra IDとの紐付けに使用する。社員マスタリストではGID列名で統一管理する。

# 2. システムアーキテクチャ

## 2.1 技術スタック

| コンポーネント | 技術/サービス | 用途 |
|-------------|-------------|------|
| フロントエンド | Power Apps（キャンバスアプリ） | 申請画面、評価画面、閲覧画面 |
| データストア | SharePoint Lists | 申請データ、マスタデータ、評価データ、メンバーデータ |
| ワークフロー | Power Automate | メール通知、ステータス遷移 |
| ファイル管理 | SharePoint ドキュメントライブラリ | 添付ファイル（改善前画像・改善後画像・その他資料）の保管 |
| 認証/ID | Microsoft Entra ID | ユーザー認証、GID（SonyID）との紐付け |

## 2.2 データフロー

Power Apps → SharePoint Lists（データ登録/更新） → Power Automate（トリガー: リスト項目の作成/変更時） → Outlook（メール通知）の一方向フローを基本とする。評価結果による条件分岐はPower Automate内で処理し、褒賞金額に応じて次の承認者を動的に決定する。表彰区分が改善提案以外の場合はスコアリングをスキップする分岐も含む。
