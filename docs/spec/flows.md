# 5. Power Automate フロー設計

## 5.1 フロー一覧

> ※ シンプルプランでは承認完了時の通知は申請者と承認者のみ。

| No. | フロー名 | トリガー | 処理概要 |
|-----|---------|--------|---------|
| 1 | 申請通知フロー | Power Appsから直接起動 `[v2]` | 課長へ承認依頼メール送信。課長=申請者の場合は部長へ送信。1人目評価者なし（§2）の場合は部長へ直接遷移 |
| 2 | 課長承認フロー | Lists項目変更時（課長評価完了） | 承認: 金額判定→部長通知 or 完了メール（申請者+課長） / 差戻: NG通知 |
| 3 | 部長承認フロー | Lists項目変更時（部長評価完了） | 承認: 申請者+部長へ完了メール / 差戻: NG通知 |
| 4 | 取下げ通知フロー `[v2]` | Power Appsから直接起動（§4） | 承認者・回覧者へ取下げ通知メール。取消前ステータスに応じて通知先を判定 |
| 5 | リマインダーフロー `[v2]` | スケジュール（日次 9:00 JST）（§7） | 承認・回覧が滞留している案件のCurrentAssigneeEmailへ督促メール送信 |
| 6 | ステージング転送フロー `[v2]` | Power Appsインスタントフロー（§1） | ステージングリストの添付ファイルをドキュメントライブラリに転送 |
| 7 | ステージングクリーンアップフロー `[v2]` | スケジュール（日次）（§1） | 48時間経過した未提出ステージングレコードを自動削除 |
| 8 | 回覧通知フロー `[v2]` | Lists項目変更時（回覧者リスト ReviewStatus=承認）（§3） | 次の回覧者へ通知、または全回覧者承認後に評価者へ遷移 |
| 9 | 回覧差戻フロー `[v2]` | Power Appsインスタントフロー（§3） | 回覧者差戻時にメインリストのステータス更新＋申請者へ差戻通知メール送信 |
| 10 | 申請通知フロー（再提出） `[v2]` | Power Appsインスタントフロー（§5） | 下書き・差戻・取下げからの再提出時に申請通知フローをPower Appsから直接起動。初回提出と再提出で同一フロー（No.1）を使用 |

## 5.2 申請通知フロー詳細

| ステップ | アクション | 詳細 |
|---------|-----------|------|
| 1 | トリガー: Power Appsから `[v2]` | パラメータ: RequestID。§5によりPower Appsトリガーに変更（初回提出・再提出共通） |
| 2 | メイン項目取得 | RequestIDで改善提案メインリストをLookUp |
| 3 | 承認者取得 | 社員マスタから申請者GID（SonyID）でLookUpし、課長GID→課長メールを取得。課長本人フラグ=1（課長=申請者）の場合は部長GIDを取得 |
| 3a | 条件分岐: 1人目評価者なし? `[v2]` | ApproverManagerが空（§2: 1人目なし）の場合→部長へ直接遷移、ステータスを「部長評価中」に更新 |
| 3b | 条件分岐: 課長=申請者? | 課長本人フラグ=1の場合→部長へ承認依頼メール（課長承認スキップ）、それ以外→課長へ承認依頼メール |
| 4 | メール送信 | 承認依頼メール（提案タイトル、申請者名、リンク付き） |

> **v2変更点（§2）**: 1人目の評価者がなし（ApproverManager=Blank）の場合、部長へ直接遷移しステータスを「部長評価中」に更新する。
>
> **v2変更点（§5）**: トリガーを「Lists項目作成時」から「Power Appsから直接起動」に変更。RequestID採番はPower Apps側で実施済み（KZ形式）。下書き・差戻・取下げからの再提出時もbtnSubmit.OnSelectから直接起動される。
>
> **v2変更点（§7）**: 課長=申請者で部長へ直接遷移する分岐で、CurrentAssigneeEmailを部長のメールに更新、EvaluationStartDateを`utcNow()`に更新する。

## 5.6 メールリンクURL設定 `[v10.1]`

メールテンプレートに含まれるPower Appsリンクの設定仕様。各メールの用途に応じてリンク先画面とURLパラメータを使い分ける。

| テンプレート | リンクテキスト | リンク先画面 | URLパラメータ |
|-------------|--------------|-------------|-------------|
| 3-1_申請通知_承認依頼.html | 評価画面を開く | 評価画面 | `?RequestID={RequestID}&EvalType=課長` |
| 3-2_課長承認_部長へ承認依頼.html | 評価画面を開く | 評価画面 | `?RequestID={RequestID}&EvalType=部長` |
| 3-2_課長承認_承認完了.html | 申請内容を確認する | 閲覧画面 | `?RequestID={RequestID}` |
| 3-2_課長承認_差戻通知.html | 申請内容を修正する | 申請フォーム | `?RequestID={RequestID}&Mode=Edit` |
| 3-3_部長承認_承認完了.html | 申請内容を確認する | 閲覧画面 | `?RequestID={RequestID}` |
| 3-3_部長承認_差戻通知.html | 申請内容を修正する | 申請フォーム | `?RequestID={RequestID}&Mode=Edit` |
| 3-4_取下げ通知.html `[v2]` | 申請内容を確認する | 閲覧画面 | `?RequestID={RequestID}` |
| 3-5_リマインダー.html `[v2]` | 対象の申請を確認する | Power Apps（App.OnStartで動的判定） | `?RequestID={RequestID}` |

>
> **注意**: `{AppID}` はPower AppsアプリのGUID。アプリ公開後に確定するため、テンプレートのプレースホルダー `XXXXXXXX` をGUIDに置換する。パラメータの `{RequestID}` はPower Automateの動的コンテンツで差し込む。
>
> **v2補足**: リマインダー（3-5）はEvalTypeパラメータを含めない（§7 DJ-7: App.OnStartで動的判定）。取下げ通知（3-4）も閲覧画面への遷移のみ。

### 5.7 v2 追加フロー `[v2]`

#### 5.7.1 ステージング転送フロー（No.6）（§1）

**トリガー**: Power Apps（手動トリガー）

**入力パラメータ**:

| 変数名 | 型 | 説明 |
|--------|---|------|
| RequestID | string | 改善提案のRequestID |
| StagingBeforeID | string | 改善前ステージングレコードのID（未使用の場合は空文字） |
| StagingAfterID | string | 改善後ステージングレコードのID（未使用の場合は空文字） |
| StagingOtherID | string | その他ステージングレコードのID（未使用の場合は空文字） |

**処理フロー**:

| ステップ | アクション | 詳細 |
|---------|-----------|------|
| 1 | トリガー: Power Appsから | パラメータ: RequestID, StagingBeforeID, StagingAfterID, StagingOtherID |
| 2 | 各ステージングIDのループ | StagingBeforeID, StagingAfterID, StagingOtherIDを順に処理（空でないIDのみ） |
| 3 | 添付ファイルの取得 | ステージングレコードの添付ファイルを取得 |
| 4 | ドキュメントライブラリにコピー | RequestID + FileCategory付きでファイルを保存 |
| 5 | ステージングレコード削除 | 転送完了後にステージングレコードを削除 |

> **詳細設計**: `powerautomate/flow-staging-transfer-build.html`

#### 5.7.2 ステージングクリーンアップフロー（No.7）（§1）

**トリガー**: スケジュール（繰り返し: 毎日1回）

**目的**: 提出・下書き保存されずに放置されたステージングレコードを自動削除する。

**処理フロー**:

| ステップ | アクション | 詳細 |
|---------|-----------|------|
| 1 | トリガー: Recurrence | 毎日1回実行 |
| 2 | 対象レコード取得 | 「添付ファイルステージング」リストから作成日時が48時間以上前のレコードを取得（ODataフィルタ: `Created lt '@{addHours(utcNow(), -48)}'`） |
| 3 | Apply to each | 取得したレコードを1件ずつ削除 |

> **設計判断**: 48時間の猶予により長時間の入力作業中にレコードが消えるリスクを回避。空レコード・ファイル付きレコードを区別せず一律削除。
>
> **詳細設計**: `powerautomate/flow-staging-cleanup-build.html`

#### 5.7.3 回覧通知フロー（No.8）（§3）

**トリガー**: Lists項目変更時（回覧者リスト、条件: ReviewStatus = "承認"）

**処理フロー**:

| ステップ | アクション | 詳細 |
|---------|-----------|------|
| 1 | トリガー: 項目が変更されたとき | 回覧者リスト、条件: ReviewStatus = "承認" |
| 2 | 変更アイテム取得 | トリガーのレコードからRequestID、ReviewStatus、ReviewOrderを取得 |
| 3 | 次の回覧者を検索 | 回覧者リストから `RequestID = 同一 AND ReviewOrder > 現在のOrder AND ReviewStatus = "未回覧"` で昇順ソート、先頭1件を取得 |
| 4 | 条件分岐: 次の回覧者がいる? | |
| 4a | Yes → 次の回覧者へ通知 | 回覧依頼メールを送信。CurrentAssigneeEmail = 次の回覧者のメール、EvaluationStartDate = `utcNow()` に更新 `[§7]` |
| 4b | No → 全回覧者承認完了 | メインリストのステータスを「課長評価中」に更新（§2: 1人目評価者なしの場合は「部長評価中」）。課長（または部長）へ承認依頼メールを送信。CurrentAssigneeEmail = 評価者のメール、EvaluationStartDate = `utcNow()` に更新 `[§7]` |

> **差戻処理**: 差戻はPower Appsからインスタントフロー（回覧差戻フロー No.9）を直接呼び出して処理する。回覧通知フローは承認時のみトリガーされる。
>
> **回覧全承認後の通知**: 回覧通知フロー内で直接課長（または部長）への承認依頼メールを送信する（方式1: フロー内で完結）。
>
> **§2との連携**: 全回覧者承認後、ApproverManagerが空の場合は「部長評価中」に遷移しApproverDirectorに通知する。

#### 5.7.4 回覧差戻フロー（No.9）（§3）

**トリガー**: Power Appsから（Power Appsトリガー）

**入力パラメータ**:

| 変数名 | 型 | 説明 |
|--------|---|------|
| RequestID | string | 改善提案のRequestID |
| ReviewerGID | string | 差戻を行った回覧者のGID |
| DismissComment | string | 差戻コメント |

**処理フロー**:

| ステップ | アクション | 詳細 |
|---------|-----------|------|
| 1 | トリガー: Power Appsから | パラメータ: RequestID, ReviewerGID, DismissComment |
| 2 | メイン項目取得 | RequestIDで改善提案メインリストを取得 |
| 3 | メインリスト更新 | ステータスを「差戻」に更新、CurrentAssigneeEmail = 空にクリア `[§7]` |
| 4 | メール送信 | 申請者へ回覧差戻通知メール（差戻コメント付き） |

> **差戻コメント**: 回覧者リストにはコメント列を持たない（§3 DJ-7）ため、Power Appsからフローパラメータとして渡す。

#### 5.7.5 取下げ通知フロー（No.4）（§4）

**トリガー**: Power Appsから（Power Appsトリガー）

**入力パラメータ**:

| 変数名 | 型 | 説明 |
|--------|---|------|
| RequestID | string | 改善提案のRequestID |
| CancelReason | string | 取消理由（任意、空の場合あり） |
| PreviousStatus | string | 取消前のステータス |

**処理フロー**:

| ステップ | アクション | 詳細 |
|---------|-----------|------|
| 1 | トリガー: Power Appsから | パラメータ: RequestID, CancelReason, PreviousStatus |
| 2 | 項目の取得 | 改善提案メインリストからRequestIDで取得 |
| 3 | 通知先の判定 | PreviousStatusパラメータに基づき通知先を決定 |
| 4 | メール送信 | 通知先へ取下げ通知メール（取消理由付き） |

**通知先判定ロジック**:

| PreviousStatus | 通知先 | 備考 |
|----------------|--------|------|
| 申請中 | ApproverManager（空ならApproverDirector） | §2: 1人目なしの場合は2人目に通知 |
| 回覧中 | 回覧者リストから現在の回覧者（ReviewStatus=未回覧 かつ 最小ReviewOrder） | §3: 直列方式のため1名 |
| 課長評価中 | ApproverManager | — |
| 部長評価中 | ApproverDirector | — |

> **「下書き」はフロー呼び出し対象外**: Power Apps側のbtnCancel.OnSelectで `varPreviousStatus <> "下書き"` の条件分岐によりフローは起動されない。
>
> **メールテンプレート**: `powerautomate/3-4_取下げ通知.html`

#### 5.7.6 申請通知フロー（再提出）（No.10）（§5）

§5の下書き保存機能により、申請通知フロー（No.1）のトリガーが「Power Appsから直接起動」に変更されたことで、再提出時の通知も同一フローで処理される。

**起動タイミング**:
- 新規提出: btnSubmit.OnSelect内で `申請通知フロー.Run(varEditRequestID)`
- 差戻後の再提出: 同上（既存レコードの更新だが、Power Appsトリガーのためフロー起動可能）
- 取下げ後の再提出: 同上
- 下書きからの提出: 同上

> **§3統合**: 回覧者ありの場合は回覧通知フロー（No.8）が代わりに起動される。`If(CountRows(colReviewers) > 0, 回覧通知フロー.Run(...), 申請通知フロー.Run(...))` の分岐をbtnSubmit.OnSelectで実行。

#### 5.7.7 リマインダーフロー（No.5）（§7）

**トリガー**: Recurrence（スケジュール）

| 項目 | 内容 |
|------|------|
| 実行間隔 | 1日、毎朝 9:00 JST |
| タイムゾーン | (UTC+09:00) 大阪、札幌、東京 |

**処理フロー**:

| ステップ | アクション | 詳細 |
|---------|-----------|------|
| 1 | トリガー: Recurrence | 毎日 9:00 JST |
| 2 | 複数の項目の取得 | 改善提案メインリスト。ODataフィルタ: `Status eq '回覧中' or Status eq '課長評価中' or Status eq '部長評価中'` |
| 3 | Apply to each | ステップ2の結果を1件ずつ処理 |
| 3-1 | 条件分岐: リマインド対象か？ | `addDays(items('Apply_to_each')?['EvaluationStartDate'], 2) < utcNow()` |
| 3-1a | Yes → メール送信 | CurrentAssigneeEmailのメールアドレス宛に督促メール送信 |
| 3-1b | No → スキップ | EvaluationStartDateから2日以内のためスキップ |

**リマインダー条件**:
- `CurrentAssigneeEmail` が空でない（承認待ち/回覧待ちの案件）
- `EvaluationStartDate + 2日 < 現在日時`（承認依頼から48時間経過後の最初の9:00から送信）
- 条件に該当する限り毎朝送信（間引きなし）

> **メールテンプレート**: `powerautomate/3-5_リマインダー.html`
>
> **経過日数の算出**: `div(sub(ticks(utcNow()), ticks(items('Apply_to_each')?['EvaluationStartDate'])), 864000000000)` で日数を計算。
